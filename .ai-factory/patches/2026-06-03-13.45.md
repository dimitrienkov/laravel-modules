# Octane paths-config review follow-ups: dead code, eager validation, backup resolve, naming

**Date:** 2026-06-03 13:45
**Files:** src/Application/Support/ModuleDirectoryPaths.php, src/Registry/ModuleDirectoryScanner.php, src/Support/ModulePathsConfig.php, src/Support/ModuleConfigKeys.php, src/Providers/ModuleLoaderServiceProvider.php, tests/Feature/OctaneWorkerLifecycleTest.php, tests/Feature/ModuleLoaderServiceProviderTest.php, tests/Unit/Application/Support/ModuleDirectoryPathsTest.php, docs/octane.md, docs/configuration.md
**Severity:** low

## Problem

Review of the `feature/octane-compatibility` branch (centralizing `modules.paths.*` into `ModulePathsConfig`) surfaced a cluster of quality issues, none blocking:

- An unreachable `if ($roots === [])` branch in `ModuleDirectoryPaths::configuredRoots()` — `ModulePathsConfig` already guarantees a non-empty list.
- `ModulePathsConfig` classdoc claimed "validates eagerly at the composition root", but the singleton resolved lazily on first path-service `make()`, weakening Octane fail-fast.
- `ModuleDirectoryPaths::backupRoot()` returned the raw config value, while `ModuleStatePaths::root()` ran it through `PathNormalizer::resolveAbsolute()` — a relative `modules.paths.backup` would resolve against CWD instead of base path.
- Triple-duplicated `$directories` docblock; an inaccurate "Octane anti-pattern" comment on `Repository` injection.
- `ModuleConfigKeys::STATE`/`BACKUP` and `readRequiredString()` named by type, not entity (they are root paths).
- Test gaps: no backup-root freeze test under Octane, no broken-`state`/`backup` rejection symmetry; a brittle test read the private `ModuleRegistry::snapshot` via reflection.
- `docs/octane.md` had a grammatically broken `--max-requests` sentence.

## Root Cause

These are the predictable residue of a "move the source of truth" refactor: when reading/validation is hoisted into a new owner (`ModulePathsConfig`), the *consumers* keep stale copies of checks (empty-list branch), the new owner's doc gets aspirational ("eager") before the wiring catches up, and a sibling code path (`backupRoot`) is left on the old raw-value behavior because it wasn't the focus of the change. The reflection-based test was testing an implementation detail (snapshot identity) rather than the observable freeze contract, so it added fragility without unique coverage.

## Solution

- **Dead code:** removed the unreachable empty-list branch; deleted the now-stale `configuredRootsRejectsEmptyList` test (coverage stays in `ModulePathsConfigTest`).
- **Eager validation:** force `$this->app->make(ModulePathsConfig::class)` at the top of the provider `boot()`; added `bootFailsFastWhenPathsConfigIsBroken` feature test; kept the classdoc eager wording now that it is true.
- **Backup resolve:** `backupRoot()` now uses `PathNormalizer::resolveAbsolute(..., $this->basePath)` (idempotent on absolute paths); added relative + absolute unit tests.
- **Ownership boundary (no new abstraction):** kept `realpath` (scanner, touches FS) vs pure-string (`ModuleDirectoryPaths`) deliberately separate; documented the split in classdocs instead of extracting a shared guard.
- **Naming:** `readRequiredString` → `readRequiredPath`; `ModuleConfigKeys::STATE`/`BACKUP` → `STATE_ROOT`/`BACKUP_ROOT`. Left `configured*` constructor params as the intentional "raw config before resolve" name.
- **Tests:** added backup-root freeze test; added broken-`state`/`backup` rejection symmetry via a shared helper; removed the reflection snapshot test (freeze contract already covered by `featureValuesAreFreshWhileEnabledStaysFrozenWithinOneWorker`).
- **Docs:** fixed the `--max-requests` sentence; noted backup relative-path resolution.

No `[FIX]` runtime logging added (package core forbids it). Gate: `rector` → `format` → `phpstan` (0 errors) → `test` (44 arch / 606 unit / 133 feature green); `format:dry` and `rector:dry` clean.

## Prevention

- After hoisting read/validation into a new owner VO, grep consumers for now-dead copies of the same check and remove them in the same change.
- A docblock that says "eager" must be backed by an actual eager resolve (a `make()` in `boot()`), or the word must not be there — comments are contracts.
- When refactoring one path getter, check sibling getters (state vs backup) resolve identically; asymmetric `resolveAbsolute` usage is a latent relative-path bug.
- Prefer observable-behavior tests over reflection into private fields; only keep a reflection test if it locks a contract nothing else covers.
