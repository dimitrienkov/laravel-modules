## Problem

Lifecycle use cases (install, update, scaffold, remove) could leave modules in a partially-applied state when persistence failed after filesystem operations succeeded. State validation was too lenient (silently coercing invalid settings shapes). CLI naming used double-negative (`--no-backup`). No architecture guard prevented raw filesystem I/O from leaking into use case layer.

## Root Cause

1. No compensation/rollback logic after copy/replace operations succeeded but manifest/state writes failed.
2. `ModuleStateRepository::extractValues()` silently replaced invalid `settings` and `settings.values` with empty arrays instead of throwing.
3. `RemoveModuleUseCase` deleted state before directory in permanent-delete mode — if directory delete failed, state was already lost.
4. `ScaffoldModuleUseCase` with `--force` skipped registry uniqueness check entirely, allowing name collisions across configured roots.
5. Skeleton creation and provider stub rendering mixed into use case class with unchecked `file_get_contents`/`file_put_contents`.
6. String literal source types (`'directory'`/`'zip'`) instead of typed enum; skipped values had no reason diagnostics.

## Solution

- **Install**: wrapped manifest/state writes in try/catch after `copyDirectory()`; on failure, cleanup target directory and state, rethrow as `ModuleInstallException` with original as previous.
- **Update**: saved existing state document before replacement; wrapped manifest/state writes after `replaceDirectoryWithBackup()`; on failure, restore from backup directory and rewrite original state document.
- **Remove**: reordered permanent delete to directory-first then state; renamed `$noBackup` to `$deletePermanently` and CLI option to `--delete-permanently`.
- **Scaffold**: always check registry uniqueness even with `force`; allow `force` only when existing module points to same target path; extracted skeleton creation to `ModuleSkeletonBuilder` with checked I/O; added compensation block that cleans up on any failure.
- **State repository**: strict validation of `settings` and `settings.values` shapes; checked `delete()` verifies unlink/rmdir results; `extractValues()` throws `InvalidModuleStateException` for non-object shapes.
- **Naming**: `sourceType: string` → `sourceKind: ModuleSourceKind` enum; `list<string> $skippedValues` → `list<SkippedFeatureValue>` with `key` and `reason`.
- **Architecture**: added file I/O boundary test with narrow allowlist of infrastructure classes.
- **Documentation**: corrected all docs and README to reflect immutable `module.json` + mutable `state.json` separation.

## Prevention

- Architecture test guards against raw `file_get_contents`/`file_put_contents`/`unlink`/`rmdir` leaking back into use case or repository orchestration code — only allowlisted infrastructure classes may use direct I/O.
- Compensation patterns in use cases: any post-copy/post-replace persistence failure triggers cleanup/rollback before re-throwing typed exception.
- Strict state validation catches corrupt `state.json` early with actionable error messages instead of silent degradation.
- Typed enum and diagnostic DTO prevent string-literal drift and preserve failure context for debugging.
