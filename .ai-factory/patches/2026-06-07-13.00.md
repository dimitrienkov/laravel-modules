# MoonShine feature form dropped sibling overrides and silently coerced bad input

**Date:** 2026-06-07 13:00
**Files:** src/MoonShine/Support/FeatureValueWriter.php, src/MoonShine/Pages/ModuleFormPage.php
**Severity:** high

## Problem

Two blocking defects plus one follow-up in the MoonShine admin feature-flags form:

1. `FeatureValueWriter::write()` rebuilt the persisted explicit value set from an
   empty array, then handed it to `ModuleStateRepository::writeValues()` as a full
   replacement. A partial form submit (e.g. posting only `retries`) silently
   dropped every other existing explicit override (`driver=file` vanished) — a
   field absent from the request was indistinguishable from an intentional delete.
2. `FeatureValueWriter::coerce()` coerced permissively before the strict
   normalizer ran: `filter_var($v, FILTER_VALIDATE_BOOLEAN)` turned an unknown
   token like `"maybe"` into `false`, and `is_numeric($v) ? (int) $v` truncated
   `"3.5"` to `3` and `"1e2"` to `1`. Invalid transport values were persisted as
   plausible-but-wrong data instead of being rejected.
3. `ModuleFormPage::optionRules()` built the enum membership rule as the string
   `in:' . implode(',', $options)`. Laravel splits that on commas, so an enum
   option that itself contained a comma was validated as two separate values.

## Root Cause

1. The writer treated the form submit as the complete desired state. But a
   MoonShine form only carries the fields it rendered/posted; it is a partial
   patch, not a snapshot. Replacing the whole value set with a partial patch
   deletes anything not in the patch. The writer never read the current explicit
   values to merge onto.
2. PHP's lenient coercion (`FILTER_VALIDATE_BOOLEAN` without
   `FILTER_NULL_ON_FAILURE`, `is_numeric` + `(int)` cast) fails open: it produces
   a value for almost any input. The strict VO normalizer downstream can only
   reject what reaches it unchanged, but the lenient coercion had already
   rewritten bad input into something the normalizer accepts.
3. String-concatenated validation rules inherit the framework's parameter
   delimiter semantics. `in:a,b` is two values by construction; any option value
   containing the delimiter is mis-parsed.

## Solution

1. `write()` now seeds `$explicit` from `state->readValues($module)->explicitValues()`
   and applies each submitted schema key pointwise: a valid override sets only its
   key, a cleared non-boolean field / a value equal to the schema default
   `unset()`s only its key, and keys absent from the submit are left untouched.
   Persistence still flows solely through `writeValues()`.
2. Coercion now fails closed. `coerceBool()` accepts real bools, maps an
   absent/off switch (`null`) to `false`, and uses
   `filter_var($v, FILTER_VALIDATE_BOOLEAN, FILTER_NULL_ON_FAILURE) ?? $v` so an
   unrecognised token passes through unchanged for the normalizer to reject.
   `coerceInt()` accepts real ints and `filter_var($v, FILTER_VALIDATE_INT)`
   strings, passing fractional/exponent strings through untouched.
3. `optionRules()` returns `Rule::in($definition->options)` (array-based), keeping
   each option intact regardless of commas. PHPDoc widened to `string|In`.

No runtime logging was added — per project convention the diagnostic surface is
typed `InvalidManifestException`s from the normalizer plus regression tests.

## Prevention

- **Admin form writers persist a partial patch, not a snapshot.** Always read the
  current explicit/mutable state and merge the submitted fields onto it before
  calling a full-replacement persistence method. Absent field != delete.
- **Transport coercion must fail closed before a strict VO normalizer.** Prefer
  `FILTER_NULL_ON_FAILURE` / `FILTER_VALIDATE_INT` over `FILTER_VALIDATE_BOOLEAN`
  bare and `is_numeric` + cast. Coercion should only narrow unambiguous encodings;
  anything else must reach the normalizer unchanged so it can reject it.
- **Never build Laravel validation rules by string concatenation when a value can
  contain the delimiter.** Use `Rule::in()` / rule objects so option values stay
  intact.
- Regression coverage: partial-submit preservation (unit + feature), clear-one-
  keep-others, invalid bool/int rejection via `InvalidManifestException`, and an
  enum option containing a comma staying a single `Rule::in()` entry.
