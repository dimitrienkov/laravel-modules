# Token-grep falsely reported an "unused" test parameter that a positional caller relied on

**Date:** 2026-06-01 18:49
**Files:** tests/Support/CreatesSourceArchive.php (reverted), tests/Unit/Application/Support/ModuleSourcePreparerTest.php
**Severity:** medium

## Problem

While executing a review-fix plan (extract `Version`/`ModuleGroup` VOs, plus
nits), one planned step said to delete a "confirmed unused" parameter
`$extraEntries` from the `zipModuleSource()` test helper. The plan certified it
unused via `grep extraEntries tests/` → 0 hits. After applying it, the full
suite went red:

```
ModuleSourcePreparerTest::prepareThrowsWhenSourceContainsStateFile
Failed asserting that exception of type "ModuleSourceException" is thrown.
```

That test injects a forbidden `state.json` into a source archive to assert the
preparer rejects it — a real shipped-module invariant ("source modules must not
carry state.json").

## Root Cause

The "unused" verdict came from grepping the **parameter name token**, but the
sole caller passed the argument **positionally**:

```php
$this->zipModuleSource($this->tempDir . '/with_state.zip', $manifest,
    ['state.json' => json_encode(['enabled' => true])]); // 3rd positional arg
```

A positional call never contains the substring `extraEntries`, so the token
grep returned 0 and the parameter looked dead. It was load-bearing — removing it
silently dropped the extra zip entry, so the archive no longer contained
`state.json` and the guard had nothing to reject.

## Solution

Reverted that single step: kept `$extraEntries` (and clarified its docblock to
name the `state.json` rejection scenario). Recorded the deviation in the plan
rather than checking the step off. The other ~24 steps (VO extraction,
consumers, docs, tests, nits) stood and the gates went green:
`rector` → `format` → `phpstan` (level max) → `test` (arch 36 / unit 520 /
feature 64).

## Prevention

- To prove a function parameter is unused, grep the **function's call sites**
  (`grep "funcName("`) and inspect each call's **arity/positional args** — never
  trust a grep for the parameter name alone. Named-argument calls contain the
  token; positional calls do not.
- A plan's "confirmed by grep" claim is an assertion to re-verify, not a fact.
  When a removal is cheap to test, run the suite immediately after that single
  change rather than batching it into a large pass.
- When removing a test helper's capability, first ask what behaviour each
  caller's extra input exercises — here it guarded a security/packaging
  invariant, which is exactly the kind of thing that must not silently lapse.
- Correctness over plan fidelity: when a planned step rests on a wrong premise,
  skip it and document why instead of forcing it through.
