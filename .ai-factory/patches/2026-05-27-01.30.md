# Code review fixes for lifecycle branch — 8 issues

**Date:** 2026-05-27 01:30
**Files:** src/Application/UseCases/ScaffoldModuleUseCase.php, src/Application/UseCases/RemoveModuleUseCase.php, src/Application/UseCases/UpdateModuleUseCase.php, src/Application/UseCases/ListModulesUseCase.php, src/Application/DTOs/ListModulesResult.php, src/Console/Commands/Modules/ModulesOptimizeCommand.php, src/Console/Commands/Modules/ModulesOptimizeClearCommand.php, src/Console/Commands/Modules/ModulesRemoveCommand.php, src/Console/Commands/Modules/ModulesListCommand.php
**Severity:** medium

## Problem

Code review of `feature/lifecycle-usecases-commands` found 8 issues: unchecked `tryDeleteDirectory()` result in scaffold force-mode, no compensation in `removePermanently()` when directory deletion fails after state deletion, narrow catch type in optimize commands, manual path normalization instead of `PathNormalizer`, missing backup cleanup after successful update, double confirmation in remove command, no UseCase for list command, and incomplete exception chain in backup-mode double-fault.

## Root Cause

Fast incremental development of lifecycle code. Each issue is individually minor but collectively creates inconsistency in error handling, convention compliance and architecture uniformity:

1. **Ignored `tryDeleteDirectory()` result** — `tryDeleteDirectory()` silently swallows failure; when followed by `skeletonBuilder->build()`, an existing directory causes a misleading error.
2. **No compensation in `removePermanently()`** — "delete state first, then directory" follows the recoverable-first rule, but lacked a fallback to restore state if directory deletion failed.
3. **Incomplete exception chain** — In backup-mode double-fault, `$restoreError` was only included as text in the message, losing the throwable from the chain.
4. **Manual path normalization** — `rtrim(str_replace(...))` instead of the project's `PathNormalizer::normalize()`.
5. **Narrow catch in optimize commands** — `ModuleCacheWriteException` instead of `ModuleExceptionInterface` like all other commands.
6. **Double confirmation in remove** — `ConfirmableTrait` + explicit confirm caused two prompts in production; cancelled removal returned SUCCESS.
7. **No UseCase for list** — Registry injected directly into command with filtering logic in the command body.
8. **Missing backup cleanup** — `UpdateModuleUseCase` left backup directories after successful updates.

## Solution

1. Replaced `tryDeleteDirectory()` with `deleteDirectory()` in a separate try-catch before the main block; wraps failure in `ModuleScaffoldException`.
2. Added state document preservation before deletion and compensation try-catch in `removePermanently()` — restores state on directory failure, handles double-fault with nested try/catch.
3. Changed double-fault exception chain to `ModuleRemoveException(previous: $restoreError)` with `$e->getMessage()` in the message text.
4. Replaced manual normalization with `PathNormalizer::normalize()`.
5. Changed both optimize commands from `catch (ModuleCacheWriteException $exception)` to `catch (ModuleExceptionInterface $e)`.
6. Removed `ConfirmableTrait` and `--force`, changed cancel exit code to `FAILURE`.
7. Created `ListModulesUseCase` + `ListModulesResult` DTO; command becomes thin adapter.
8. Added `tryDeleteDirectory($backupPath)` after successful update.

## Prevention

- When catching narrow exception types, check if other commands in the same family use a broader marker interface — maintain consistency.
- When a multi-step operation deletes state before infrastructure, always consider: "what if the infrastructure step fails?" Add compensation or at minimum document the orphan risk.
- Use existing utility classes (`PathNormalizer`, etc.) for operations that have a canonical helper.
- Commands should be thin adapters over UseCase classes — any logic beyond input mapping and output formatting belongs in the UseCase layer.
- For `final` classes in tests: mock `Illuminate\Filesystem\Filesystem` (non-final) and wrap in `LocalFilesystem` to test failure paths without bypassing finality.
