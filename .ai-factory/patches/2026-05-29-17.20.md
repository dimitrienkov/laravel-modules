# Source-origin review fixes: stale provenance on update, late group validation, dead nullable checksum

**Date:** 2026-05-29 17:20
**Files:** src/Manifest/Parsing/ManifestFieldReader.php, src/Manifest/VO/ManifestMeta.php, src/Application/UseCases/ScaffoldModuleUseCase.php, src/Application/Support/PreparedSource.php, src/Application/UseCases/UpdateModuleUseCase.php, src/Application/UseCases/InstallModuleUseCase.php, src/Manifest/VO/ModuleOrigin.php, .ai-factory/ARCHITECTURE.md, .ai-factory/DESCRIPTION.md (+ 4 test files)
**Severity:** medium

## Problem

Branch review of the meta-group / source-origin feature surfaced a cluster of issues:

- **Stale provenance on update.** `modules:update foo new.zip` left `source.checksum` equal to the hash of the *previous* archive while `installed_version` was already the new version — an internally inconsistent `source` section. A scaffold-created module (`kind=local`) updated from a zip kept `kind=local`. Modules with no prior `source` section never got one on update.
- **Asymmetric, late `--group` validation.** `--kind` was validated immediately in the command, but `--group` was only checked deep inside `ManifestMeta::fromArray` during `writeManifest`, *after* the skeleton was built — caught by the validator and rolled back, but with a generic message and wasted filesystem work.
- **Ad-hoc group regex.** A `GROUP_PATTERN` constant lived locally in `ManifestMeta` instead of beside the shared `MODULE_NAME_PATTERN` in `ManifestFieldReader`, violating the project rule "identifier validation uses the shared regex".
- **Dead nullable checksum.** `PreparedSource::$checksum` was `?string = null` with a `?? ''` fallback at the install call site, even though the only producer always sets a non-null hash.
- **Cosmetic `ksort` in `ModuleOrigin::toArray()`** that re-sorted keys inserted in a deliberate-looking order — a phantom order with no effect, inconsistent with sibling VOs.
- **Documentation drift** — docs showed a `sha256:`-prefixed checksum (code stores bare 64-char hex) and referenced a non-existent `modules.paths.groups` config key (actual key is top-level `modules.groups`).

## Root Cause

- **Provenance:** `update` used `$origin?->withInstalledVersion($version)`, a method that bumped only the version and carried the old `kind`/`checksum` forward. The freshly-computed `$prepared->checksum` was discarded. The mental model "`source` = where the current installed bytes came from" was never enforced — the code treated origin as a version stamp, not a provenance record.
- **Late validation:** validation lived only in the manifest-parsing path (`fromArray`), but scaffold builds `ManifestMeta` through the constructor, which bypasses it. The invariant had no fail-fast entry point at the use-case boundary, unlike `name` which already had `validateName()`.
- **Dead fallback:** the nullable type was wider than reality. The single producer (`ModuleSourcePreparer::prepareFromZip`) throws if `hash_file` fails, so checksum is always a non-null string — but the type permitted null, forcing a meaningless `?? ''` at the call site.

## Solution

- Added `ManifestFieldReader::assertModuleGroup(?string, field, path)` holding the single `MODULE_GROUP_PATTERN`. Both `ManifestMeta::fromArray` and a new `ScaffoldModuleUseCase::validateGroup()` (called right after `validateName`, before any path/skeleton work) delegate to it. Null returns early ("no group"); empty string is rejected — symmetric with empty `--kind=`, no silent normalization to null.
- `UpdateModuleUseCase` now writes `ModuleOrigin::forZip($candidate->meta->version, $prepared->checksum)` — fresh provenance reflecting the bytes just installed. `withInstalledVersion()` (now unused) was deleted.
- `PreparedSource::$checksum` is now a required `string`; the `?? ''` at `InstallModuleUseCase` was removed — the type guarantees non-null into `forZip(string)`.
- `ModuleOrigin::toArray()` writes keys in explicit logical order (`kind`, `installed_version`, optional `checksum`) with no `ksort`, matching `ManifestMeta::toArray`.
- Docs: removed `sha256:` prefix, fixed `modules.paths.groups` → `modules.groups`, aligned the example key order with the new deterministic output.

## Prevention

- **Provenance VOs must be rewritten wholesale, not field-patched, when the underlying artifact changes.** A "with-one-field" mutator on a provenance object is a smell — it lets stale sibling fields (checksum, kind) survive an operation that replaced the bytes they describe. When the bytes change, recompute the whole origin.
- **Every manifest invariant needs a fail-fast entry point at each real input boundary.** A constructor that bypasses the named-constructor's validation means CLI/programmatic inputs skip the check. Mirror the `validateName` pattern: validate at the use-case before touching the filesystem.
- **Keep identifier patterns in one shared reader** (`ManifestFieldReader`); a local `*_PATTERN` const in a VO is a duplication red flag during review.
- **Make types as narrow as the single producer allows.** If the only constructor always sets a value, the field is non-nullable — a nullable type invites dead `?? default` fallbacks that would write garbage if ever reached. (Echoes prior patches: prefer explicit errors over silent coercion.)
- **Drop phantom ordering.** If output is sorted afterwards (`ksort`), the insertion order is misleading dead intent — either rely on the sort or insert in final order, not both.
- **Doc examples are code too:** a checksum format prefix or a config key path in docs should be grep-checked against the implementation when the feature lands.
