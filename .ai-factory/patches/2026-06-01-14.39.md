# Strict source-origin parsing and double-fault-safe zip staging cleanup

**Date:** 2026-06-01 14:39
**Files:** src/Manifest/VO/ModuleOrigin.php, src/Application/Support/ModuleSourcePreparer.php, tests/Unit/Manifest/VO/ModuleOriginTest.php, tests/Unit/Application/Support/ModuleSourcePreparerTest.php, docs/cli.md, docs/architecture.md, docs/manifest.md, AGENTS.md
**Severity:** medium

## Problem

Post-review of `feature/meta-group-source-origin` surfaced three non-blocking but
real defects plus doc drift:

1. `ModuleOrigin::fromArray()` validated required `source` keys but accepted any
   unknown extra key silently. On the next `toArray()` write those keys were
   dropped — silent normalization that hides malformed `state.json` from the user.
2. `ModuleOrigin::parseChecksum()` used `isset($data['checksum'])`, so an explicit
   JSON `null` checksum was treated as *absent*. For `local` it was silently
   accepted; for `zip` it produced a misleading "checksum is required" error.
3. `ModuleSourcePreparer::prepareFromZip()` deleted the temp dir inside `catch`
   with an unguarded `deleteDirectory()`. If that delete threw, the cleanup
   exception replaced the primary manifest/source validation error.
4. Docs (`cli.md`, `architecture.md`, `manifest.md`, `AGENTS.md`) still promised
   `directory/zip` install/update even though the runtime is zip-only, and
   `composer rector:dry` flagged nullable-object checks in `ModuleOrigin`.

## Root Cause

- **Unknown keys:** the contract was enforced positively (required keys present)
  but never negatively (no foreign keys), so the allow-list was implicit and
  unenforced.
- **null checksum:** `isset()` conflates "key missing" with "key present but
  null". A strict contract over JSON-decoded arrays must use
  `array_key_exists()` to distinguish an explicit null from an absent key.
- **Double-fault:** a `catch` block that performs I/O cleanup is itself a
  throwing context; without a nested guard the secondary failure wins and the
  root cause is lost (the project forbids runtime logging in package core, so
  the exception chain *is* the only diagnostic).
- **Doc drift:** docs described an earlier roadmap (directory sources) instead of
  the implemented zip-only runtime.

## Solution

- `fromArray()`: compute `array_diff(array_keys($data), ['kind','installed_version','checksum'])`
  first and throw `InvalidModuleStateException` listing the unknown keys.
- `parseChecksum()`: `isset(...)` → `array_key_exists('checksum', $data)`. Now an
  explicit `null` is "present": for `local` it fails as a forbidden checksum, for
  `zip` it falls through to the existing `is_string` guard and fails as
  non-string (not as "required").
- `prepareFromZip()`: wrap the temp-dir `deleteDirectory()` in a nested
  `try/catch (Throwable)` so cleanup failure never masks the primary `throw $e`.
- Converted nullable-object checks to `instanceof Checksum` form to satisfy
  Rector's `FlipTypeControlToUseExclusiveTypeRector`.
- Rewrote the directory/zip doc references to zip-only; kept `--directory` as the
  *target root* option (not a directory source input).
- Tests: unknown-key rejection, explicit-null-checksum for local and zip, and a
  double-fault test where `deleteDirectory()` deletes then throws, asserting the
  primary validator exception (`assertSame`) propagates.

## Prevention

- For VOs hydrated from external JSON, enforce a **closed** key set (allow-list +
  explicit rejection), not just required-field checks.
- Prefer `array_key_exists()` over `isset()` whenever an explicit `null` must be
  distinguished from a missing key.
- Any `catch` block that does filesystem/cleanup work must wrap that work in a
  nested `try/catch` to preserve the original exception (double-fault guard) —
  matches the project's `runtime`/`lifecycle` rules and the no-runtime-logging
  policy where the exception chain is the diagnostic.
- When a feature narrows runtime behavior (directory → zip-only), sweep docs +
  AI context in the same change so they describe actual runtime, not roadmap.
