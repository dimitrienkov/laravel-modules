# PHPStan level max: mixed/array-key fixes after raising analysis level

**Date:** 2026-06-01 15:59
**Files:** src/Loaders/FactoryLoader.php, src/Loaders/RouteLoader.php, src/Support/LocalFilesystem.php, src/Manifest/Parsing/ManifestFieldReader.php, src/Manifest/ModuleStateRepository.php, src/Manifest/VO/FeatureSchema.php, src/Manifest/VO/FeatureValues.php, src/Registry/VO/CachedModuleDescriptor.php, src/Registry/VO/ModuleRegistryCachePayload.php
**Severity:** medium

## Problem

Raising PHPStan from `level: 8` to `level: max` (with `treatPhpDocTypesAsCertain: true`)
surfaced 13 errors of three kinds:

1. `return.type` / `argument.type`: a value validated only by `is_array(...)` is typed
   `array<int|string, mixed>` but flowed into a sink declared `array<string, mixed>`.
2. `cast.string` / `return.type` on `mixed`: a `mixed` value (config entry, JSON field,
   external resolver result) used as a `string` without narrowing.
3. `function.alreadyNarrowedType`: `is_string($key)` guards reported as "always true"
   because the surrounding `@param array<string, mixed>` made PHPStan treat the key as
   certainly a string.

## Root Cause

The common root is that PHP's `is_array()` and `!array_is_list()` do **not** prove
string keys: `json_decode('{"1":"x"}', true)` yields `[1 => 'x']` — a non-list array with
an **integer** key. So `array<string, mixed>` annotations on JSON-derived data were
optimistic lies. At level 8 PHPStan tolerated the gap; at level 9/10 it is exact about
`mixed` and key types, and `treatPhpDocTypesAsCertain: true` made it trust (and then
flag) those PHPDoc types.

Kind 3 is the same lie seen from the other side: the `is_string($key)` runtime guards in
`FeatureSchema`/`FeatureValues` are genuine defenses against integer JSON keys, but the
`@param array<string, mixed>` told PHPStan they were dead code.

## Solution

Fix the contract, never silence the checker:

- **Enforce string keys at JSON-object boundaries** (real hardening, makes the declared
  `array<string, mixed>` true via a rebuild loop PHPStan can narrow):
  `ManifestFieldReader::requiredObject()`, `ModuleStateRepository` (new private
  `requireStringKeys()` reused by `readStateFile`/`extractOrigin`/`assertJsonObject`),
  and `CachedModuleDescriptor::fromCacheEntry()` for the cached manifest.
- **Make over-tight `@param` honest** where the method only does keyed reads or already
  guards keys internally: widen to `array<array-key, mixed>` on
  `FeatureSchema::fromArray`, `FeatureValues::fromArray`,
  `ModuleRegistryCachePayload::fromCachedArray`, `CachedModuleDescriptor::fromCacheEntry`.
  This keeps the `is_string($key)` guards alive as legitimate runtime defense.
- **Narrow `mixed` explicitly** instead of casting: `RouteLoader` prefix
  (`is_string($v) ? $v : 'api'`) and string-keyed/non-null rebuild of route attributes;
  `FactoryLoader` returns the previous resolver result only `if (is_string($resolved))`,
  else falls back to the default class resolver; `LocalFilesystem::directories()` filters
  `is_string` before `array_values` to guarantee `list<string>`.

## Prevention

- Treat `is_array($x)` as `array<array-key, mixed>`, not `array<string, mixed>`. JSON
  objects with numeric-string keys decode to integer PHP keys — enforce string keys
  explicitly when the type or downstream code assumes them.
- When `treatPhpDocTypesAsCertain: true` flags a guard as "always true", check whether the
  upstream `@param`/`@return` is honest before deleting the guard. Often the type is the
  bug, not the guard — widen the type and keep the defense.
- Prefer an explicit rebuild loop over an inline `/** @var array<string, mixed> */` cast
  when the goal is a *true* string-keyed array from untrusted input — the cast is a lie,
  the loop is enforcement.
- `level: max` auto-tracks the highest level, so PHPStan upgrades can introduce new errors
  without a config change; pin a numeric level if reproducibility across versions matters.
