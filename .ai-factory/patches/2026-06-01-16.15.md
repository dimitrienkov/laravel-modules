# Removed load-bearing-lie `@var array<string, mixed>` casts over validated manifest input

**Date:** 2026-06-01 16:15
**Files:** src/Manifest/VO/Module.php, src/Manifest/ManifestSettingsValidator.php, src/Manifest/ManifestValidator.php
**Severity:** low

## Problem

Five inline `/** @var array<string, mixed> $x */` casts sat over manifest data that
PHPStan had already typed honestly. Two failure shapes:

- **Redundant restatement.** `$metaRaw`/`$settings` came straight out of
  `ManifestFieldReader::requiredObject()`, which already returns `array<string, mixed>`
  (it rebuilds the array rejecting non-string keys). The cast just repeated the inferred type.
- **Carrying lie.** `$schemaRaw` was `$settingsRaw['schema'] ?? []` after an `is_array`
  guard, i.e. `array<array-key, mixed>` — its keys can genuinely be `int`
  (`"schema": [1,2,3]`). The cast asserted `array<string, mixed>`, narrowing a type that
  is not actually guaranteed. Such a cast silences any future PHPStan error downstream that
  depends on the real (possibly int-keyed) shape — everything below it trusts the lie.

## Root Cause

The casts were left over from when the receivers (`FeatureSchema::fromArray`,
`ManifestMeta::fromArray`) required `array<string, mixed>` and the readers had looser
return types. After the contract was tightened — `requiredObject()` now honestly returns
`array<string, mixed>`, and `FeatureSchema::fromArray` now accepts `array<array-key, mixed>`
and validates keys itself — the casts became either pure noise or an unsound narrowing that
no longer matched what the code guarantees.

## Solution

Deleted all five `@var` comment lines (no runtime change — PHPDoc only):

- `Module.php:49` `$metaRaw` (redundant), `:51` `$schemaRaw` (lie)
- `ManifestSettingsValidator.php:29` `$schemaRaw` (lie)
- `ManifestValidator.php:44` `$metaRaw` (redundant), `:48` `$settings` (redundant)

The real inferred types now flow to the receivers untouched: `requiredObject()` →
`array<string, mixed>` satisfies `ManifestMeta::fromArray`; the `?? []` + `is_array` →
`array<array-key, mixed>` is accepted by `FeatureSchema::fromArray`, which re-validates keys
at runtime. `composer phpstan` (level max + Larastan) stays green, proving each cast was
removable rather than load-bearing.

Left in place intentionally: `FeatureSchema.php:43` `$definition`. Its receiver
(`FeatureDefinitionFactory::fromArray`) still requires `array<string, mixed>`, and removing
the cast honestly would cascade a `array<array-key, mixed>` contract change through the whole
`ManifestFieldReader` reader family (`optionalInt`/`optionalString`/...) — a contract refactor,
not a localized fix. The cast there is also immediately backed by `assertAllowedKeys`, which
rejects any non-allow-listed (incl. int) key on the next line.

## Prevention

- An inline `@var array<string, mixed>` over a value the type system already inferred is a
  smell: it is either redundant (delete it) or it is narrowing a type the code does not
  guarantee (a "load-bearing lie" that suppresses future errors). Prefer fixing the
  producer/consumer contract so the honest type flows, over papering it with a cast.
- When a function validates keys/shape itself at runtime, type its `@param` as the widest it
  truly accepts (`array<array-key, mixed>`), not the narrowed shape the caller wishes were true.
- Reach for `array<array-key, mixed>` (not `array<string, mixed>`) whenever the source is raw
  JSON-decoded data that has not yet had its keys proven to be strings.
- For PHPDoc-only changes the regression guard is the static gate, not a runtime test:
  `composer phpstan` going green after removing a cast is the proof it was not load-bearing.
