# Registry snapshot VO and optimize use case extraction

**Date:** 2026-05-26 21:30
**Files:** src/Registry/VO/ModuleRegistrySnapshot.php, src/Registry/ModuleRegistrySnapshotBuilder.php, src/Manifest/ModuleRegistry.php, src/Application/DTOs/OptimizeModulesResult.php, src/Application/DTOs/ClearModulesOptimizeCacheResult.php, src/Application/UseCases/OptimizeModulesUseCase.php, src/Application/UseCases/ClearModulesOptimizeCacheUseCase.php, src/Console/Commands/Modules/ModulesOptimizeCommand.php, src/Console/Commands/Modules/ModulesOptimizeClearCommand.php, src/Contracts/ModuleRegistryCacheInterface.php, src/Providers/ModuleLoaderServiceProvider.php
**Severity:** medium

## Problem

ModuleRegistry held snapshot state as two nullable arrays (`$modules`, `$orderedModules`) with array-shape protocol. `writeCache()` was a public method on concrete registry, causing optimize commands to inject `ModuleRegistry` directly instead of going through application/use case boundary. `modules:optimize-clear` injected both concrete `ModuleRegistryCache` and `ModuleRegistry`.

## Root Cause

After earlier refactoring that extracted `ModuleDirectoryScanner` and `ModuleRegistryCache`, the snapshot representation and optimize orchestration were not promoted to their own typed boundaries. The registry still owned scan logic and cache-write orchestration, mixing infrastructure concerns with runtime query responsibility.

## Solution

1. Introduced `ModuleRegistrySnapshot` as a typed immutable VO encapsulating module map + load order with deterministic order guarantee.
2. Extracted `ModuleRegistrySnapshotBuilder` — takes scanner, manifests, sorter and produces a fresh snapshot from filesystem.
3. Simplified `ModuleRegistry` to hold `?ModuleRegistrySnapshot`, removed `scan()` and `writeCache()`.
4. Added `cachePath()` to `ModuleRegistryCacheInterface`.
5. Created `OptimizeModulesUseCase` and `ClearModulesOptimizeCacheUseCase` with typed result DTOs.
6. Updated commands to inject use cases instead of concrete classes.
7. Added architecture test guards for optimize command dependencies.
8. Applied Rector cleanup across codebase.

## Prevention

- When extracting infrastructure from a service class, also extract the orchestration that commands call — don't leave "command helper" methods on domain objects.
- Architecture tests for command dependency boundaries catch regressions early.
- When adding typed snapshot VOs, `all()` should return load order (not random map order) to preserve determinism.
