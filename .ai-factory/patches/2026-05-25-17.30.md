# Atomic write deduplication — extract shared AtomicFileWriter

**Date:** 2026-05-25 17:30
**Files:** src/Support/AtomicFileWriter.php, src/Support/AtomicJsonWriter.php, src/Registry/ModuleRegistryCache.php, src/Exceptions/AtomicWriteException.php, tests/Unit/Support/AtomicFileWriterTest.php
**Severity:** low

## Problem

`ModuleRegistryCache::writeAtomically()` and `AtomicJsonWriter::write()` implemented identical atomic file writing logic independently: lock acquisition, temp file creation, content write with flush, permission preservation, atomic rename, and cleanup in finally block. 95%+ code duplication across ~80 lines each.

## Root Cause

The two classes were developed at different times for different concerns (JSON manifest writing vs PHP cache writing) without recognizing the shared atomic write pattern. The only actual difference was the exception type thrown and the content encoding (JSON vs raw PHP string).

## Solution

Extracted `AtomicFileWriter` — a generic `final readonly` class that handles the full atomic write lifecycle (directory creation → lock → temp file → write → flush → permissions → rename → cleanup). Both consumers now delegate to it via composition with default constructor parameter (`new AtomicFileWriter()`), catching `AtomicWriteException` and wrapping into their domain-specific exceptions (`ManifestWriteException` / `ModuleCacheWriteException`). No DI registration changes needed — PHP 8.1 "new in initializers" handles default creation.

## Prevention

- When implementing file I/O patterns (atomic write, lock, temp file), check if a shared utility already exists in `src/Support/`
- The `Support/` directory is the canonical location for cross-cutting utilities
- Prefer composition with default parameters over inheritance for sharing infrastructure logic
