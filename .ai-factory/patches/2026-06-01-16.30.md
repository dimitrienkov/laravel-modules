# Remove `Routes/api/*.php` convention — make RouteLoader fully config-driven

**Date:** 2026-06-01 16:30
**Files:** src/Loaders/RouteLoader.php, config/modules.php, tests/Unit/Loaders/RouteLoaderTest.php, docs/configuration.md, docs/module-structure.md, docs/architecture.md, .ai-factory/ARCHITECTURE.md, .ai-factory/DESCRIPTION.md, .ai-factory/ROADMAP.md
**Severity:** medium

## Problem

`RouteLoader` carried two competing route-loading conventions: the config-driven
`modules.routing.types.<type>` → `Routes/<type>.php` loop, plus a separate built-in
scan of `Routes/api/*.php` (`versionedApiRoutes()`) that prefixed files with
`api/<version>`. The second branch could not apply a per-version middleware group
(it always inherited the `api` type's attributes), so versioning via
`Routes/api/v1.php` and versioning via a config type were two ways to do the same
thing — one of them strictly less capable. The default config also leaked Laravel
internals into the package by hard-coding `SubstituteBindings::class` and
`ConvertEmptyStringsToNull::class` in `api.middleware`.

## Root Cause

Convention drift: a special-case scan was bolted onto a system that already had a
general, declarative mechanism. The versioned branch predated (or grew alongside)
the config-type loop and was never folded back into it, leaving a redundant,
less-flexible code path and a private `versionedApiRoutes()` helper that duplicated
glob/sort logic the type loop didn't need. The middleware leak came from copying a
host-app default into a reusable package config instead of letting the host own its
HTTP-kernel internals.

## Solution

Deleted the `versionedApiRoutes()` method and its loop in `routeFiles()`, so
`RouteLoader` is now purely `foreach (array_keys($types) as $type)` →
`Routes/<type>.php`. Versioning becomes an ordinary flat type: the host declares
`api_v1` (`prefix => 'api/v1'`, `middleware => ['api_v1']`) → `Routes/api_v1.php`,
and registers the `api_v1` middleware group in its own `bootstrap/app.php`. Trimmed
`api.middleware` to `['api']`, dropped the two framework `use` imports, and added a
commented `api_v1` example profile. Rewrote the loader unit test to assert
config-driven loading of `web` + `api_v1` (no `Routes/api/` subdirectory), with
group order following config-key order. Synced docs and aif-context artifacts
(configuration, module-structure, architecture, DESCRIPTION, ROADMAP) to the
single config-driven model.

## Prevention

- When a declarative/config-driven mechanism exists, resist adding a parallel
  built-in scan for a "special case" — model the special case as another config
  entry. Two ways to express one concept is a design smell, not a feature.
- Reusable-package configs should not hard-code the host framework's HTTP-kernel
  internals (`SubstituteBindings`, `ConvertEmptyStringsToNull`); leave those to the
  host's named middleware groups.
- When deleting a private helper, confirm no now-unused `use` imports or dangling
  helpers remain — `phpstan` at level max catches this; run it after the cut.
- Order-sensitive loader behavior should be asserted against config-key order in a
  test (here: group registration order == `modules.routing.types` key order).
