# Unify named parameter style for $previous in exception factories

**Date:** 2026-05-26 15:33
**Files:** src/Exceptions/InvalidModuleStateException.php, src/Exceptions/ModuleStateWriteException.php
**Severity:** low

## Problem

Code review found 3 exception factory methods using positional `new self("...", 0, $previous)` while all other 12+ exception classes consistently used `new self("...", previous: $previous)`. Functionally identical but stylistically inconsistent.

## Root Cause

The two exception classes were added during the lifecycle refactoring batch. The author used the positional form out of habit, not noticing the project had already adopted named parameters for `$previous` everywhere else. No linter rule catches this — it's a pure style consistency issue.

## Solution

Replaced `0, $previous` with `previous: $previous` in all 3 factory methods:
- `InvalidModuleStateException::forPath()` (line 15)
- `InvalidModuleStateException::forModule()` (line 20)
- `ModuleStateWriteException::forPath()` (line 15)

## Prevention

- When adding new exception classes, check existing exceptions for the project's parameter style before writing factory methods.
- During code review, grep for `0, $previous` in `src/Exceptions/` — any hit is a style violation.
- Consider a custom PHPStan rule or rector rule that enforces named `previous:` in RuntimeException subclass constructors.
