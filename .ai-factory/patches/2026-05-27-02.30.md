# Remove duplicate loadOrder() method from ModuleRegistryInterface

**Date:** 2026-05-27 02:30
**Files:** src/Contracts/ModuleRegistryInterface.php, src/Manifest/ModuleRegistry.php, src/Registry/VO/ModuleRegistrySnapshot.php, src/Loaders/Pipeline/ModuleLoaderPipeline.php, src/MoonShine/MoonShineModuleAutoloader.php, src/Application/UseCases/OptimizeModulesUseCase.php, src/Registry/ModuleRegistrySnapshotBuilder.php
**Severity:** low

## Problem

`ModuleRegistryInterface` defined two methods — `all()` and `loadOrder()` — that returned identical data. `ModuleRegistrySnapshot` implemented both with the same body. Callers were split: use cases used `all()`, pipeline/MoonShine used `loadOrder()`.

## Root Cause

The two methods were introduced with different semantic intent: `all()` for generic "get all modules" queries, `loadOrder()` for pipeline ordering. In practice, the snapshot stores modules in topological order from construction, so both always return the same data. The distinction was never meaningful at the implementation level.

## Solution

Removed `loadOrder()` from the interface, both implementations, and the snapshot. Kept `all()` as the single method — it's the more natural name for generic queries, and the ordering guarantee comes from the snapshot's construction (TopologicalSorter), not from the method name. Renamed snapshot internal property from `$loadOrder` to `$modules` and `$modules` (map) to `$moduleMap` for clarity.

## Prevention

- When an interface has two methods with identical signatures and implementations, question whether the semantic distinction is load-bearing. If both methods always return the same data by construction, keep one.
- Prefer method names that describe _what_ is returned (`all`) over _how_ it's ordered (`loadOrder`), when ordering is an invariant of the data structure itself.
