# Module-aware generators: view-namespace leak, parent-delegation, single-source layers

**Date:** 2026-06-02 20:50
**Files:** src/Console/Commands/Make/{MakeMail,MakeComponent,MakeController,MakeMigration,MakeFactory,Make{UseCase,Action,Query,Dto,Vo}}.php, src/Console/Concerns/ModuleAwareGenerator.php, src/Console/Support/ModuleResolver.php, src/Support/{ModuleLayer,ModuleLayout}.php, src/Application/Enums/ScaffoldComponent.php, src/Providers/ModuleLoaderServiceProvider.php, src/Loaders/{Factory,Observer,Policy}Loader.php, tests/**
**Severity:** medium

## Problem

Code review of the `make:* --module` generators (green gates, risk 🟡) found four
functional risks plus maintainability debt:

- Generated mail/component classes referenced views by **host** name
  (`markdown: 'mail.digest'`, `view('components.alert')`). The Blade file was
  written into the module, but the runtime reference resolved against host views
  the `ViewLoader` never registered → view-not-found or a silent wrong-view hit.
- `MakeController::buildFormRequestReplacements()` re-implemented the parent's
  whole 10-token map and hardcoded `App\Http\Requests` — a Laravel token rename
  would silently strand request imports in the host.
- The generator shadowing relied on an undocumented Laravel detail; the rebind
  invariant test asserted only 7 of 21 commands, so a regression wouldn't fail.
- One layer string (`Application/UseCases`, …, `Domain/VO`) was spelled out in
  three places, and `ModuleLayout` carried ~12 getters with zero `src/` callers.

## Root Cause

- **View leak:** native `*MakeCommand::buildClass()` stamps the *bare* view from
  `getView()`, while the module's views are registered under the `{module->name}`
  namespace. The relative Blade path and the in-class reference are two different
  concerns; only the reference needed a namespace, not the file path.
- **Controller duplication:** the parent computes the request namespace as an
  inline literal with no override seam, so the subclass copied the whole method
  instead of adjusting the one host-specific value.
- **Weak invariant:** the test hardcoded a subset of pairs instead of deriving
  them from the provider's own `GENERATORS` constant.
- **Scattered constants:** no single owner for the architectural-layer strings.

## Solution

- **F1:** override only `buildClass()` in `MakeMail`/`MakeComponent`; call the
  parent, then rewrite the single quoted view reference to `{module->name}::view`.
  Never touch `getView()`/`writeView()`/`viewPath()` (those build the file path —
  a `::` there would corrupt it). Inline components are skipped.
- **F2:** let the parent build the full token map, then `array_map` a targeted
  `str_replace($this->rootNamespace().'Http\\Requests' → ModuleLayout::requestsNamespace($module))`
  over the values. The request *files* are already routed into the module by the
  trait's `call()` override, so only the namespace string needed repointing. The
  module namespace comes from one `ModuleLayout` helper, not a literal.
- **M5:** a string-backed `Support\ModuleLayer` enum backs each layer's relative
  directory; the 5 architectural generators read `namespaceSegment()`,
  `ScaffoldComponent` reads `relativeDirectory()`. Only the two methods with real
  call-sites were added (no speculative `directory()/namespace()` — that would
  just relocate the dead getters the task removed).
- **M8:** `Console\Support\ModuleResolver` (final readonly) centralises
  `Str::snake(trim(...))` normalisation and `has()`-before-`find()` resolution;
  the trait and `MakeMigration` both delegate to it.
- **F4:** the rebind test now derives the full pair list from `GENERATORS` via
  Reflection (+ `MigrateMakeCommand`), so a newly shadowed command is covered
  automatically.

## Prevention

- When a generator writes a file **and** embeds a reference to that file, treat
  the path and the reference as separate concerns. Override the class-body
  builder (`buildClass`) for the reference; never repurpose the path builder.
- Prefer delegating to the parent generator and adjusting its output over copying
  its token map — a copied token set is upgrade-hostile across Laravel releases.
- Pin "shadowing/rebind" invariants to the **source list** (a constant read via
  Reflection), not a hand-maintained subset, so coverage can't silently drift.
- A magic string with domain meaning that appears in 2+ places gets one owner
  (enum/VO/constant); but only add accessors that have real call-sites — moving
  dead getters into a new type is not consolidation.
- Resolve modules with `ModuleRegistryInterface::has()` then `find()` rather than
  `try { find() } catch` — existence check over flow-by-exception.
