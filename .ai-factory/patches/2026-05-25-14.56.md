# Runtime loader lifecycle and production cache reliability

**Severity:** high

## Problem

Несколько runtime paths были ненадёжны в реальном Laravel 13 lifecycle:

- console command/route loaders слушали concrete kernel через plain `afterResolving()` и пропускали уже resolved console kernel;
- resource loaders eager-resolve'или `migrator`, `translator`, `view` и `BladeCompiler` через constructor DI;
- `FactoryLoader` перезаписывал host application's factory resolver;
- MoonShine bridge регистрировался в `register()` и не срабатывал для already resolved core;
- production cache писался напрямую в `bootstrap/cache/modules.php`;
- loader pipeline репортил raw exception без module/loader context;
- docs/context продолжали показывать устаревшие `bool/int/string`, `save()` и list-form dependencies.

## Root Cause

Реализация смешала обычный container resolution с package-provider lifecycle. Laravel package helpers используют `ServiceProvider::callAfterResolving()` semantics: зарегистрировать future callback и выполнить его сразу, если service уже resolved. Без этой immediate ветки loaders работали в unit happy path, но ломались в console bootstrap, где kernel уже resolved до boot package providers.

В cache path проблема была в предположении, что single `file_put_contents()` достаточно для PHP cache. Для production cache это неверно: partial write может оставить syntactically broken PHP file, который затем попадёт в `require`.

Документационный drift возник после breaking-clean API pass: код уже закрепил `getBool/getInt/getString`, `saveValues()` и object-only dependencies, но README/docs/AI-context всё ещё описывали старые контракты.

## Solution

Добавлен `ContainerLifecycleHooks` с package-style `callAfterResolving()` semantics. Console loaders теперь слушают `Illuminate\Contracts\Console\Kernel`, проверяют concrete `Illuminate\Foundation\Console\Kernel` перед вызовом `addCommandPaths()`/`addCommandRoutePaths()` и регистрируют paths через `$app->booted()`.

Resource loaders переведены на deferred registration без eager service resolution. `FactoryLoader` сохраняет previous factory resolver и делегирует non-module models в host resolver или Laravel default resolver. MoonShine hook перенесён в provider `boot()` и работает для future и already resolved core.

`ModuleRegistryCache::write()` теперь пишет через lock, temp file, full byte count, `fflush()` и atomic `rename()`. `load()` заворачивает failed `require` в `InvalidModuleCacheException`. Pipeline reports теперь используют `ModuleLoaderException` с `loaderClass`, `moduleName`, `modulePath` и previous exception.

Добавлены regression tests для lifecycle timing, factory resolver preservation, cache atomic behavior, enriched pipeline reporting, invalid loader tag diagnostics и optimize-clear registry reset. README/docs/AGENTS/DESCRIPTION/ARCHITECTURE синхронизированы с фактическим API.

## Prevention

- Для Laravel package integrations использовать package-style immediate-if-resolved hooks, а не plain `afterResolving()`.
- Не инжектить тяжёлые deferred framework services в loader constructors, если Laravel package helper обычно регистрирует их через lifecycle callback.
- При установке глобального Laravel static resolver сохранять previous resolver и делегировать unrelated input.
- Production PHP cache писать только через temp+flush+rename под lock; failed `require` всегда оборачивать в typed exception с path context.
- После breaking-clean API changes сразу запускать grep по README/docs/AI-context на старые method names и legacy manifest forms.
