# Diagnostic logging review remediation: event taxonomy, terminal failure events, ERROR gating, Throwable propagation

**Date:** 2026-06-02 12:21
**Files:** src/Contracts/ModuleDiagnosticsInterface.php, src/Support/Logging/{ModuleLogger,NullModuleDiagnostics}.php, src/Application/UseCases/{Install,Update,Remove,Scaffold,Enable,Disable,Optimize,ClearModulesOptimizeCache}ModuleUseCase.php, src/Registry/ModuleRegistryCache.php, src/Loaders/{Migration,Command,Factory,Lang,View,Event,BladeComponent,ServiceProvider,Route}Loader.php, src/Loaders/Pipeline/ModuleLoaderPipeline.php, src/Loaders/VO/{SkipReason,LoadReport,PipelineRunSummary}.php, src/Application/Enums/LifecycleOperation.php, src/Support/ModuleLayout.php, docs/{logging,feature-toggles,cli}.md, + 11 test files
**Severity:** medium

## Problem

A branch review (self + 10 `dandy-*` sub-agents) of the opt-in diagnostic-logging
feature found a cluster of correctness-adjacent issues, no hard blockers:

- Inconsistent lifecycle event-name taxonomy: `lifecycle.{op}.rolledBack` (camelCase)
  and a static `lifecycle.backup.created` next to otherwise dotted snake_case names.
- No terminal failure event: a throw after `lifecycleStarted` left a "hanging"
  `started` with no matching terminal, so logs could not answer "did this finish?".
- ERROR events (`pipeline.loader.failed`) were gated by the same per-category toggle
  (`events.pipeline`) as DEBUG noise — silencing the noise also silenced failures.
- The caught `Throwable` was reduced to a class string + message in
  `pipeline.loader.failed` / `cache.invalid`, losing trace and the `$previous` chain.
- `substr($dir, strlen($module->path) + 1)` duplicated across 7 loaders.
- `MODULES_LOG_LEVEL` threshold lookup was case-sensitive → `INFO` fell through to
  the debug fallback.
- Doc/docblock drift: `source=local` (no such `ModuleSourceKind`), a `Log::channel`
  facade in prose (the package forbids facades), an orphaned `logging.md` breadcrumb,
  a wrong `SkipReason::EmptyDirectory` emitter list, an unanchored `§2.1` reference.

## Root Cause

The taxonomy strings were authored ad hoc per call site with no single vocabulary
rule, so camelCase and a static name slipped in. The "exactly one terminal per
started" invariant was never modelled: `started` was emitted eagerly but the failure
paths only emitted intermediate `rolledBack` markers and re-threw, with no terminal.
The `shouldLog` gate applied the category toggle uniformly, treating an operator's
"mute this category" as "mute everything in it" — including failures the operator
almost certainly still wants. The lost-cause issue came from preferring a "clean"
scalar context (`$exception::class`) over the richer Throwable object that PSR-3
channels already know how to render.

## Solution

- **One terminal per started, uniformly.** Wrapped the post-`started` body of all 8
  lifecycle UseCases in `try { … } catch (Throwable $e) { lifecycleFailed($op,
  $module, $e); throw $e; }`. The catch re-throws the *same* object, so exception
  type/message/`$previous` are unchanged and the double-fault root cause (the
  wrapped exception whose `$previous` is the original error) is preserved. Existing
  inner rollback/restore catches stay as intermediate `rolled_back` markers.
  Pre-`started` guard rejections (e.g. `assertCanRemove`, already-enabled) stay
  *before* `started` and deliberately emit nothing.
- **Taxonomy + levels:** `rolled_back`, `backup_created` (now INFO, no `operation`
  key), new `lifecycle.{op}.failed` (ERROR, carries the Throwable), and
  `pipeline.finished` escalates DEBUG→WARNING when `failed > 0`.
- **ERROR severity-bypass:** in `shouldLog`, events at ERROR+ skip the per-category
  toggle but still honour the global threshold.
- **Throwable in context:** `loaderFailed`/`cacheInvalid`/`lifecycleFailed` log the
  object under `exception` (+ a readable `message`).
- **`ModuleLayout::relativeToModule()`** replaces the 7 duplicated `substr` offsets.
- **Case-insensitive threshold** via `strtolower()` in `priority()`.
- Doc/docblock sync; new tests assert names+levels+context, the bypass, the
  case-insensitive threshold, the terminal-on-failure invariant, and that
  guard rejections emit nothing.

## Prevention

- When a component emits a `started`/lifecycle event, model the invariant
  explicitly: every started path must reach exactly one terminal. The cheapest
  enforcement is a single outer `catch (Throwable) { emitTerminal; throw; }` that
  re-throws unchanged — it never alters the exception, so it composes with inner
  rollback catches.
- Operator "off switches" for log categories should never be able to hide errors —
  gate ERROR+ on the severity threshold only, not on category toggles.
- Log the `Throwable` object, not `$e::class . $e->getMessage()` — PSR-3 channels
  render trace and `$previous` from the object; the string form silently drops the
  causal chain.
- Treat log event names as a contract: keep one vocabulary (dotted snake_case here)
  and sync `docs/logging.md` in the same change that touches an emit string. A grep
  for the old name across `src/` and `docs/` should return zero.
- Normalise externally-supplied enum-like config (`MODULES_LOG_LEVEL`) before map
  lookup; a case mismatch that falls through to a default is a silent misconfig.
