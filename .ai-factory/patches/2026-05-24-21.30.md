# Architectural decomposition of runtime, manifest schema, and MoonShine integration

**Date:** 2026-05-24 21:30
**Files:** src/Manifest/*, src/Registry/*, src/Loaders/Pipeline/*, src/MoonShine/*, src/Providers/ModuleLoaderServiceProvider.php, tests/**
**Severity:** medium

## Problem

Multiple classes combined too many responsibilities, hindering schema evolution, registry cache, lifecycle commands, and optional MoonShine auto-discovery. Specifically:
- `FeatureDefinition` mixed VO, parser, normalizer, and type validation
- `ModuleRegistry` combined scanner, cache, snapshot, and sort logic
- `ModuleLoaderServiceProvider` was wiring + loader pipeline + MoonShine integration
- `MoonShineLoader` was registered via eager `$this->app->bound()` check instead of `afterResolving`
- Registry cache included mutable `settings.values`, creating stale-data risk under Octane
- Module name regex was duplicated across `ManifestMeta` and `ModuleDependencies`

## Root Cause

Architecture described a more mature package runtime than the initial MVP code. Boundaries for schema evolution, registry cache, and optional integrations were not yet extracted. MoonShine integration was implemented as an eager container-bound check when the architecture required a lifecycle hook.

## Solution

Decomposed into role-based namespaces with single-responsibility classes:
- `Manifest/Parsing/ManifestFieldReader` — shared field validation primitives + module name rule
- `FeatureType` enum, `FeatureDefinitionFactory`, `FeatureValueNormalizer` — separated from `FeatureDefinition` VO
- `Module` no longer carries `FeatureValues` — registry/loaders work with descriptor only
- `ModuleManifestRepository::readValues()` — dedicated method for fresh value reads
- `Registry/ModuleDirectoryScanner` — extracted FS scanning logic
- `Registry/ModuleRegistryCache` — extracted cache read/write/forget with version validation
- `Loaders/Pipeline/ModuleLoaderPipeline` — extracted loader orchestration from service provider
- `MoonShine/MoonShineModuleAutoloader` — dedicated runner via `afterResolving(CoreContract::class)`
- `ModuleRegistry::reset()` — explicit in-memory invalidation for lifecycle commands
- Cache payload no longer includes `settings.values`
- Removed empty `src/Services/` directory

## Prevention

- When a class grows beyond 3 responsibilities, split before adding features
- Optional integrations must use `afterResolving` or `class_exists` guards, never `$app->bound()` in `register()`
- Production cache should only include immutable discovery data; mutable runtime values belong in per-request reads
- Module name validation regex should live in exactly one place (ManifestFieldReader)
- Architecture tests should verify namespace boundaries (e.g., depth(0) for loader contract enforcement)
