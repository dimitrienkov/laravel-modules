# MoonShine admin form 500 when clearing an int/enum/string feature field

**Date:** 2026-06-06 21:08
**Files:** src/MoonShine/Pages/ModuleFormPage.php, src/MoonShine/Resources/ModulesResource.php, src/MoonShine/Support/FeatureValueWriter.php (new), src/MoonShine/Support/ModuleIndexGrouping.php (new), src/MoonShine/Data/ModuleAdminDto.php, src/MoonShine/Support/FeatureFieldFactory.php, src/MoonShine/Pages/ModuleIndexPage.php, src/MoonShine/Pages/ModuleDetailPage.php, src/Exceptions/ModuleNotFoundException.php
**Severity:** high

## Problem

In the MoonShine admin feature-flags form, clearing an `int`/`enum`/`string`
field and submitting produced an unhandled HTTP 500. `ModuleFormPage::rules()`
did not translate String `min`/`max` either, so an over-length string passed the
form and only blew up in the normalizer. The crash surfaced as an
`InvalidManifestException` whose message talked about the **manifest**, even
though the real source was form input.

## Root Cause

The page's `rules()` was **not a superset** of `FeatureDefinition::normalize()`.
A cleared field arrives as `null` (Laravel's `ConvertEmptyStringsToNull`) or `''`
(middleware off); MoonShine's `getRequestValue()` returns `?? false`, and the
resource mapped `false → null`. That `null`/`''` was passed straight into the
strict normalizer, which rightly rejects it — but nothing caught it, so the user
got a 500 instead of a graceful revert-to-default. Validation that is weaker than
the persistence-layer validator always leaks the strict error to the user.

## Solution

1. `rulesFor()` rewritten `switch → match` returning a per-type rule list, with
   String `min`/`max` translated to Laravel `min:`/`max:` (which compare string
   length — mirroring the normalizer's `mb_strlen`). Now a strict superset of
   `normalize()`.
2. Added a clear-field guard on the write side: for every non-`Boolean` type, a
   `null` or blank-trim string means "cleared" → the override is dropped (reverts
   to the schema default) instead of being normalized. `Boolean` is exempt (no
   empty state; off is a coercible `false`).
3. Hardened the surrounding asymmetries the review surfaced: typed
   `ModuleNotFoundException` (with `?Throwable $previous` + `noSelection()`) in
   `findItem()`/`save()` instead of generic `RuntimeException`; key↔column
   conversion encapsulated on `ModuleAdminDto` (`featureColumn` /
   `featureKeyFromColumn`); write side extracted to `FeatureValueWriter` and index
   ordering to `ModuleIndexGrouping`; page-route selection by closure instead of a
   `'detail'`/`'form'` string literal.

## Prevention

- When a form/controller feeds a strict validator (VO normalizer, manifest
  parser), the form rules MUST be a superset of that validator — otherwise the
  strict error leaks as an unhandled 500. Add a test that clears each field type
  and asserts a graceful outcome, not just the happy path.
- Distinguish "field absent" from "field present but cleared": a test that only
  checks key-absence does NOT prove a clear-revert. Make the request double report
  `has() === true` for a present-but-null value.
- Mirror the package's typed-exception convention at every new boundary
  (`ModuleExceptionInterface`, `?Throwable $previous`); never reach for generic
  `RuntimeException` where a typed factory exists.
