# Single validating owner for `modules.paths.*`; remove dead state-root containment check

**Date:** 2026-06-03 12:07
**Files:** src/Support/ModuleConfigKeys.php (new), src/Support/ModulePathsConfig.php (new), src/Support/ModuleStatePaths.php, src/Application/Support/ModuleDirectoryPaths.php, src/Registry/ModuleDirectoryScanner.php, src/Providers/ModuleLoaderServiceProvider.php, tests/Support/CreatesLifecycleEnvironment.php, tests/Unit/Support/ModulePathsConfigTest.php (new), tests/Feature/ModuleLoaderServiceProviderTest.php, tests/Feature/OctaneWorkerLifecycleTest.php, 11 other test files, docs/configuration.md, docs/octane.md, .ai-factory/ROADMAP.md
**Severity:** medium

## Problem

A branch review (project rules + dandy-* lenses) surfaced a cluster of
maintainability defects on `feature/octane-compatibility` — the suite was green,
but the design lied:

- `ModuleStatePaths::validate()` was dead (0 callers in `src/`/`tests/`), and the
  branch kept feeding it a live `directories` constructor argument — the
  composition root threaded validated config into a method nothing called.
- Resolving `modules.paths.*` was duplicated across 3 private provider methods
  **and** a second, divergent copy in the test helper that silently filtered
  invalid entries (`array_filter(..., 'is_string')`). Production threw on bad
  config; the lifecycle tests quietly accepted it — so the tests validated a
  contract the runtime did not have.
- Two path services carried a "structurally validated by the composition root …
  once" docblock that was false (config was read/validated 3× per boot), plus
  `?string` override properties (`$stateRoot`, `$backupRoot`) that silently fell
  back to hardcoded defaults, shadowing the `config/modules.php` defaults.
- An Octane test asserted snapshot identity that was trivially true by singleton
  memoization, under a name promising a "freeze across requests" guarantee.

## Root Cause

The original Octane DI-cleanup (passing scalars instead of capturing the config
`Repository`) was the right instinct but landed half-applied: it extracted the
*reads* into three private provider methods without giving them a single owner,
and the test harness grew its own parallel resolver instead of reusing the
production one. Two copies of "resolve + validate config" with different
strictness is the structural cause — once the copies exist, they drift, and the
weaker copy (tests) hides the stronger copy's failures. The dead `validate()`
survived because removing a constructor argument felt riskier than leaving an
unused "maybe useful later" hook — the classic live-config-into-dead-sink
anti-pattern.

## Solution

Introduced one validating value object, `ModulePathsConfig`, as the single
owner of `modules.paths.*`: a `final readonly` VO with a `fromRepository()`
named constructor that reads the `Repository` contract, validates eagerly
(non-array list, per-entry non-empty string **with index + type in the
message**, non-empty list, required non-empty `state`/`backup` strings), and
stores only scalars — never the framework object (Octane-safe). Bound as a
shared singleton, so validation runs once per worker. All three path services
(`ModuleStatePaths`, `ModuleDirectoryScanner`, `ModuleDirectoryPaths`) and the
test helper now build from this one resolver, so invalid config fails
identically in tests and at runtime. Key string literals moved into a
`ModuleConfigKeys` holder (mirrors `ModuleFileNames`). Removed the dead
`validate()` + its `directories` argument, the `?string` fallbacks (defaults
now come only from package config through the VO), and renamed override
properties to `configuredStateRoot`/`configuredBackupRoot` so "configured value"
and "resolved value" stop sharing a token. Made the Octane snapshot test honest
(renamed to `singletonMemoizesSnapshotAcrossScopeResets`, dropped the tautology).

## Prevention

- When a service takes a constructor argument, grep for the argument's only
  consumer before adding it — if the only consumer is a method with 0 callers,
  you are wiring live config into a dead sink. Delete the sink, not just silence
  the analyzer.
- Config resolution + validation belongs to exactly **one** owner (a VO/parser).
  If a test helper needs the same scalars, it must call that owner — never grow a
  second copy. A test copy with weaker validation (`array_filter` vs throw) makes
  the suite green for the wrong reason and hides the real contract.
- Per skill-context: prefer explicit typed exceptions over silent coercion;
  surface *which* entry failed (index + `get_debug_type`), not just "something is
  wrong".
- A docblock that asserts an invariant ("validated once") is not a guarantee —
  carry the invariant in a type/single owner, or the prose drifts from the code.
- Name a test for what it actually pins. If an assertion is true by object
  memoization, do not name it after a behavioural freeze guarantee a sibling test
  already owns.
