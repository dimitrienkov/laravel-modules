# Batch review fix: 9 findings across src/ — validation, scoping, perf, conventions

**Date:** 2026-05-25 00:30
**Files:** src/Manifest/FeatureValues.php, src/Manifest/Module.php, src/Manifest/ManifestValidator.php, src/Manifest/ManifestMeta.php, src/Manifest/ManifestState.php, src/Loaders/ConfigLoader.php, src/Support/ComposerNamespaceResolver.php, src/Manifest/FeatureDefinition.php, src/Support/TopologicalSorter.php, src/Console/Commands/ModulesOptimizeCommand.php, src/Console/Commands/ModulesOptimizeClearCommand.php, src/Manifest/ModuleDependencies.php, composer.json
**Severity:** medium

## Problem

Code review of 40 files in `src/` revealed 9 actionable issues:
1. `FeatureValues::fromArray()` passed hardcoded `'manifest'` to error context instead of real module name.
2. camelCase aliases (`displayName`, `installedAt`, `updatedAt`) created ambiguity in manifest format.
3. `ConfigLoader` merged module configs into global keys, allowing modules to overwrite `app.key` etc.
4. `ComposerNamespaceResolver` read+parsed+sorted `composer.json` on every `resolve()` call.
5. `strlen()` measured bytes, not characters — UTF-8 strings got wrong min/max validation.
6. `TopologicalSorter` validated dependencies of disabled modules, preventing safe disable+remove workflows.
7. Console commands lacked `final` — the only classes in the package without it.
8. `meta.name` accepted any string — no format contract for config keys, artisan commands, dependency graph.
9. `meta.dependencies` accepted names that could never match a valid `meta.name` after step 8.

## Root Cause

Multiple independent gaps accumulated during initial development:
- **Steps 1, 8, 9:** Validation boundaries weren't traced end-to-end. `meta.name` was validated as "non-empty string" but never checked for format, and that loose contract propagated to dependency names and error messages.
- **Step 2:** Backwards-compatibility sugar was added preemptively before any users existed.
- **Step 3:** ConfigLoader was ported from a pattern where modules own their config namespace, but the scoping prefix was never added.
- **Step 4:** Performance wasn't measured — `resolve()` being called N times wasn't noticed.
- **Step 5:** `strlen` is the default PHP reflex; `mb_strlen` requires conscious choice.
- **Step 6:** TopologicalSorter was designed for the "everything enabled" happy path.
- **Step 7:** Simple oversight — commands were written before the `final` convention was formalized.

## Solution

All 9 fixes applied in one pass with 30 new unit tests covering each change. Key patterns:
- **Propagate context to error messages** (step 1) — pass identifying information through the call chain, don't assume it at the call site.
- **One canonical format** (steps 2, 8, 9) — remove sugar/aliases in a pre-1.0 package; validate identifiers at parse boundary with a shared regex.
- **Namespace isolation** (step 3) — prefix module config keys with `{module_name}.{config_file}`.
- **Lazy cache** (step 4) — `??=` pattern with `private ?array` for expensive derivations on an effectively-immutable input.
- **Character-aware operations** (step 5) — `mb_strlen($v, 'UTF-8')` + explicit `ext-mbstring` in `require`.
- **Conditional validation** (step 6) — dependency validation gated on `$module->isEnabled()`; cycle detection still runs for all.

## Prevention

- When a VO is used as an identifier (config key, CLI argument, dependency name), define its format regex once and validate at every parse boundary.
- When merging into a shared namespace (config, routes), always scope under the module's identity.
- When a method reads filesystem on every call, consider whether the input changes between calls. If not, cache.
- When using `strlen` in validation, ask: "bytes or characters?" If the answer could ever be "characters", use `mb_strlen`.
- When validation runs for all items in a collection, ask: "does this item's state exempt it from some checks?"
- Run a `final` convention check in architecture tests to catch new non-final classes.
