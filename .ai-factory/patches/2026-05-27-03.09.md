# Code review findings: exception chaining, TOCTOU, unguarded read, test warning

**Date:** 2026-05-27 03:09
**Files:** src/Application/UseCases/RemoveModuleUseCase.php, src/Support/LocalFilesystem.php, src/Application/Support/PreparedSource.php, src/Application/Support/ModuleDirectoryOperations.php, src/Support/ZipExtractor.php, src/Application/Support/ModuleSkeletonBuilder.php, tests/Unit/Application/UseCases/RemoveModuleUseCaseTest.php, tests/Unit/Application/UseCases/UpdateModuleUseCaseTest.php
**Severity:** critical (exception chaining), medium (TOCTOU), low (unguarded read, test warning)

## Problem

Four issues found during code review of the lifecycle-usecases-commands branch:

1. **RemoveModuleUseCase** double-fault catch blocks passed `$restoreError` as `previous` instead of the original `$e`, losing the root cause exception in the chain.
2. **Four TOCTOU patterns** used check-then-act (`isFile()`/`isDirectory()` then `delete()`/`deleteDirectory()`) in cleanup paths, creating a race window.
3. **ModuleSkeletonBuilder::renderProviderStub()** called `filesystem->get()` between an `isFile()` check and usage without try/catch — a failure would throw raw `FileNotFoundException` instead of `ModuleScaffoldException`.
4. **UpdateModuleUseCaseTest::doubleFaultPersistAndRestoreFailure** physically deleted backup directories in a mock callback, causing `@rename()` to emit a PHP warning captured by PHPUnit.

## Root Cause

1. Copy-paste of the double-fault pattern with forgotten variable substitution — `$restoreError` was used where `$e` was intended.
2. Habitual check-then-act instead of try-then-verify — a common TOCTOU anti-pattern in filesystem operations.
3. Assumption that `isFile()` guard makes the subsequent `get()` safe — but the file can disappear or become unreadable between the two calls.
4. Test simulated failure via physical side effects (deleting backup dirs) instead of mock-based failure injection, causing real filesystem warnings.

## Solution

1. Replaced `$restoreError` with `$e` in both `removePermanently()` and `removeWithBackup()` double-fault throws. Tightened test assertions to verify the correct exception type is in the chain.
2. Eliminated all four TOCTOU patterns:
   - `LocalFilesystem::deleteFileIfExists()` — `delete() || !isFile()` (try then verify)
   - `PreparedSource::cleanup()` — removed guard, wrapped in try/catch (cleanup path)
   - `ModuleDirectoryOperations::tryDeleteDirectory()` — `deleteDirectory() || !isDirectory()` (try then verify)
   - `ZipExtractor::extractToTemp()` — removed guard, wrapped in try/catch (error cleanup)
3. Wrapped `filesystem->get()` in try/catch, converting to `ModuleScaffoldException`.
4. Added `set_error_handler(fn () => true, E_WARNING)` / `restore_error_handler()` around the execute call in the test.

## Prevention

- In double-fault catch blocks, always verify which exception variable is passed as `previous` — the original `$e` should be preserved, not the cleanup/restore error.
- For cleanup/error paths doing filesystem operations, prefer try-then-verify or try/catch over check-then-act.
- When a method does an existence check followed by a filesystem read, wrap the read in try/catch to handle the race window.
- Tests should inject failures via mocks, not via physical side effects that trigger real warnings.
