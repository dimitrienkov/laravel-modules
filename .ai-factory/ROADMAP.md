# Roadmap пакета `dimitrienkov0/laravel-modules`

> Модульная Laravel-платформа для коммерческой поставки модулей заказчикам: манифест `module.json`, loader-pipeline, фичетоглы, опциональный MoonShine admin-UI, Octane-safe. Версия `2.0` — мажорный рефакторинг ядра без обратной совместимости с `1.x`.

## Milestones

### Фаза 0 — Актуализация репозитория и тулинга (фундамент)
- [x] **Обновление composer-зависимостей** — поднять до актуальных линейок: `laravel/framework ^12|^13` (последние патчи), `moonshine/contracts ^4`, `moonshine/core ^4`, `orchestra/testbench ^10|^11`, `phpunit/phpunit ^11|^12`, `phpstan/phpstan ^2`, `larastan/larastan ^3` (заменить `nunomaduro/larastan ^2`), `rector/rector ^2`, `driftingly/rector-laravel ^2`, `friendsofphp/php-cs-fixer ^3` (последний), добавить `pestphp/pest ^3` и `pestphp/pest-plugin-arch ^3`. Запустить `composer update`, зафиксировать `composer.lock`. _Выполнено 2026-05-23: composer.lock зафиксирован, ничего не понижалось, добавлен `config.allow-plugins`. Уточнение после архитектурного решения: поддерживаются только Laravel 12/13; MoonShine и Inertia должны стать soft-зависимостями (`suggest` + `require-dev`), а не обязательным runtime `require`._
- [x] **Конфиги инструментов качества** — `phpstan.neon` (level 8, larastan, `treatPhpDocTypesAsCertain: false`, кастомное правило-запрет `Illuminate\Database\Eloquent\Model` в `Application/UseCases/*`), `rector.php` (Laravel-set + type-declaration + dead-code + early-return), `.php-cs-fixer.dist.php` (`@PSR12` + strict-types declare + ordered-imports + проектные правила), `phpunit.xml` с testsuites `Architecture`/`Unit`/`Feature`, `Pest.php` bootstrap. _Выполнено 2026-05-23: кастомное правило PHPStan вынесено TODO-якорем в Фазу 1, остальное настроено и пробутилось локально._
- [x] **Composer scripts** — `composer scripts`: `test` (массив-форма), `test:arch`, `test:unit`, `test:feature`, `phpstan`, `rector:dry`, `format`, `format:dry`. _Выполнено 2026-05-23._
- [ ] **CI workflow — отложено** — изначально планировался GitHub Actions на матрицу `php: [8.3, 8.4] × laravel: [12, 13]`, но пользователь принял решение пока не использовать CI вообще (репозиторий не публикуется на GitHub в данной фазе). Качественные гейты остаются локальными через `composer phpstan`, `composer rector:dry`, `composer format:dry`, `composer test`. Возврат к CI — отдельная задача при публикации пакета.

### Фаза 1 — v2.0 ядро (переписать уже реализованное)

- [x] **Контракты и манифест** — `LoaderInterface` (только `load(Module)` + `priority()`), `ModuleRegistryInterface`, `ModuleManifestRepositoryInterface`, `FeatureRepositoryInterface`, `NamespaceResolverInterface`, `ManifestValidatorInterface`. JSON-схема `module.json` **без секции `autoload`** (pure convention over configuration) — только `meta`, `state`, `settings.schema`, `settings.values`. `meta.dependencies` — map `moduleName => Composer SemVer constraint`, короткая форма `["users"]` нормализуется в `{"users": "*"}`. Value-объекты: `Module`, `ManifestMeta`, `ManifestState`, `ModuleDependencies`, `FeatureSchema`, `FeatureDefinition`, `FeatureValues` — все `final readonly`.
- [x] **Support-утилиты и исключения** — `ModuleLayout` (`final readonly`, единая точка истины путей и namespaces модуля: `routesDir`, `migrationsDir`, `factoriesDir`, `configDir`, `langDir`, `viewsDir`, `bladeComponentsDir`, `commandsDir`, `consoleRoutesFile`, `channelsFile`, `providersDir`, `observersDir`, `policiesDir`, `middlewareDir` + namespace-методы `bladeComponentNamespace`, `factoryNamespace`, `middlewareNamespace`, `modelNamespace`, `observerNamespace`, `policyNamespace`), `AtomicFileWriter` (shared tmp + rename + flock), `AtomicJsonWriter` (JSON-обёртка над `AtomicFileWriter`), `ApplicationNamespaceResolver` (через `Application::getNamespace()` и `Application::path()`), `ContainerLifecycleHooks` (safe `callAfterResolving` wrapper), `TopologicalSorter` для `meta.dependencies`, `ZipExtractor`, проверка version constraints через `composer/semver`, типизированные исключения (`InvalidManifestException`, `ModuleNotFoundException`, `ModuleDependencyMissingException`, `CyclicDependencyException`, `ManifestWriteException`). _`ComposerNamespaceResolver` переименован в `ApplicationNamespaceResolver` при переходе на `Application::getNamespace()`. `AtomicFileWriter` извлечён из дублирующейся логики записи в `AtomicJsonWriter` и `ModuleRegistryCache`._
- [x] **ModuleRegistry и ModuleManifestRepository** — singleton-сканер `paths.directories` с прогревом на boot worker'а, чтение/валидация/атомарная запись `module.json`, топологическая сортировка, чтение прогретого кеша из `bootstrap/cache/modules.php` одним `require` без FS-операций. Никакого mutable-state кроме private `?array $cache = null`.
- [x] **FeatureRepository (Octane-safe runtime)** — `scoped`-биндинг, per-request in-memory кеш, читает `module.json` напрямую через `ModuleManifestRepository` (не из `modules.php`-кеша) — чтобы MoonShine-правки `settings.values` подхватывались следующим request'ом без `optimize-clear`. API Фазы 1: `get/bool/int/string`; Blade directive `@feature(...)` и presentation bridge — Фаза 2; Gate/Policy bridge не входит в ядро.
- [x] **Новый ServiceProvider с pipeline** — переписать `ModuleLoaderServiceProvider`: `register()` биндит контейнер (singleton/scoped по таблице из ARCHITECTURE §13), `boot()` делает один проход `foreach loader in sortedByPriority { foreach module in loadOrder { if enabled → loader->load($module) } }`. MoonShine — только через `afterResolving(CoreContract::class)` и только при наличии класса/контракта. Inertia route integration — только при наличии Inertia в host-приложении. Удалить двойной `ConfigLoaderService::autoload()` из `register/boot`.
- [x] **Миграция уже реализованных лоадеров на новый контракт** — `ConfigLoader` (Config/*.php merge), `RouteLoader` (config-driven `modules.routing.types`, версионирование через `Routes/api/v1.php`, `Routes/api/v2.php` → prefix `api/v1`, `api/v2`), `MigrationLoader`, `FactoryLoader`, `ServiceProviderLoader` (convention `Providers/*ServiceProvider.php`, без classmap-обхода). Удалить старые `Services/*Service.php` и старый `ModuleLoaderServiceProvider`.
- [x] **MoonShineLoader без classmap-магии** — тонкая обёртка `$core->autoload($module->namespace)` для каждого включённого модуля; регистрация на `afterResolving(CoreContract::class)`. Удалить `discoverPages/discoverResources` и reflection-обходы из 1.x.
- [x] **Архитектурные тесты на ядро** — Pest+arch: `final readonly` у VO в `Manifest/`, `final` у Loader'ов и сервисов, все классы из `Loaders/` реализуют `LoaderInterface`, запрет `Illuminate\Support\Facades\*` вне `Console/Commands` и `MoonShine`, запрет `static` properties в `src/`, запрет `Illuminate\Database\Eloquent\Model` в `Application/UseCases/*`. Прогон в CI обязателен.

### Фаза 2 — v2.0 расширение (новые функции)

- [x] **Новые лоадеры** — `LangLoader` (`Lang/{locale}/*.php` через `loadTranslationsFrom`, namespace = `Str::snake($module->name)`), `ViewLoader` (`Resources/views/` через `loadViewsFrom`), `BladeComponentLoader` (`View/Components/` через `Blade::componentNamespace`), `CommandLoader` (class-based из `Console/Commands/`), `ConsoleRouteLoader` (`Routes/console.php` в актуальном Laravel-стиле: `Artisan::command(...)` + `Schedule::command(...)`), `EventLoader` (`Providers/EventServiceProvider.php` или auto-discovery), `ObserverLoader` (`Domain/Observers/` по соглашению `Model → ModelObserver`), `PolicyLoader` (`Domain/Policies/` аналогично), `MiddlewareLoader` (`Http/Middleware/` с named-регистрацией), `BroadcastLoader` (`Routes/channels.php`). _Выполнено 2026-05-25: все 10 лоадеров реализованы, каждый с unit-тестом; интегрированы в pipeline с hardening._
- [x] **Application/UseCases жизненного цикла** — `InstallModuleUseCase`, `UpdateModuleUseCase`, `RemoveModuleUseCase`, `EnableModuleUseCase`, `DisableModuleUseCase`, `ScaffoldModuleUseCase`. Все — `final readonly` с DI, без фасадов, без хелперов. _Выполнено 2026-05-25._
- [x] **Artisan-команды жизненного цикла** — тонкие обёртки над UseCase: `make:module`, `modules:list`, `modules:enable`, `modules:disable`, `modules:install <zip|path>` (валидация манифеста до распаковки, dependency check), `modules:update` (бэкап + merge `settings.values`), `modules:remove` (backup/delete, не откатывает миграции автоматически). `InstallHook` отложен до Фазы 3 (lifecycle events). _Выполнено 2026-05-25._
- [ ] **Генераторы `make:* --module`** — основной публичный API генераторов: стандартные Laravel-команды получают module-aware режим через `--module` (`make:model Post --module=Blog -mfs`, `make:controller ... --module=Blog`, `make:migration ... --module=Blog` и т.д.). Архитектурные генераторы пакета используют тот же стиль: `make:use-case`, `make:action`, `make:query`, `make:dto` + `--module`. MoonShine-генераторы не дублируются: MoonShine-артефакты создаются командами самого MoonShine. Stubs в `src/stubs/`, единый базовый класс генератора с резолвом namespace через `ApplicationNamespaceResolver`.
- [ ] **MoonShine admin-UI** — `ModulesResource` (список модулей с переключателем enabled/disabled и действиями install/update/remove на in-memory коллекции из `ModuleRegistry::all()`, без модели Eloquent), `ModuleSettingsPage` (динамическая форма по `settings.schema`: `bool→Switcher`, `int→Number` с min/max, `enum→Select`, `string→Text`), запись через `ModuleManifestRepository::updateFeatureValues(FeatureValues $values)` (никаких сырых array на границе).
- [x] **Кеш registry для production** — формат `bootstrap/cache/modules.php` с массивом сериализованных `Module` + `load_order`, интеграция с Laravel `optimize`/`optimize:clear` через `$this->optimizes()`. `ModuleRegistry` читает кеш одним `require`. `FeatureRepository` намеренно мимо кеша — читает `module.json` напрямую. _Выполнено в рамках Фазы 1: `ModuleRegistryCache`, команды `modules:optimize`/`modules:optimize-clear`, хук `$this->optimizes()` в `ModuleLoaderServiceProvider`._
- [ ] **Unit и Feature тесты** — PHPUnit + Orchestra Testbench: unit-тесты на каждый Loader (через `Module`-фабрику), `ModuleManifestRepository` (atomic write, schema validation, file locking), `TopologicalSorter` (циклы, плоский граф, изолированные модули), `ApplicationNamespaceResolver`, `AtomicFileWriter`, `AtomicJsonWriter`, `ModuleLayout`. Feature-тесты команд `modules:install/update/remove/enable/disable/list/optimize`. Feature-тесты `FeatureRepository` (повтор чтения в рамках request'а, сброс между request'ами через container scoped). MoonShine-тесты пропускаются при отсутствии MoonShine. _Значительный прогресс: 261 тест, 1207 assertions (Pest + PHPUnit). Покрыты все 15 лоадеров (включая hardening: defensive Reflection, `ContainerLifecycleHooks`, namespace resolution через `ModuleLayout`), manifest layer (VO, parsing, validators), registry, support-утилиты (`AtomicFileWriter`, `AtomicJsonWriter`, `ApplicationNamespaceResolver`, `TopologicalSorter`, `ModuleLayout`), MoonShine autoloader, feature-тесты scoped binding и optimize-команд. Всё реализованное — протестировано. Остаются только тесты на lifecycle-команды и MoonShine admin-UI (ожидают реализации самих фич)._
- [ ] **Octane-совместимость и архитектурные гарантии** — прогон под Swoole/RoadRunner/FrankenPHP с проверкой отсутствия leak'ов: state сервисов не накапливается, `FeatureRepository::$cache` сбрасывается между запросами, `ModuleRegistry` остаётся immutable после первого скана. Архитектурный тест `FeatureRepository` ≠ singleton. Документация требований к биндингам в `Octane.md`. _Частичный прогресс: scoped binding `FeatureRepository` реализован и протестирован (`FeatureRepositoryScopedBindingTest`). Остаётся прогон под Swoole/RoadRunner/FrankenPHP и `Octane.md`._
- [ ] **Документация и миграция** — `docs/architecture.md`, `docs/manifest.md`, `docs/loaders.md` (включая создание собственных), `docs/commands.md`, `docs/moonshine.md`, `docs/octane.md`, `docs/migration-from-1x.md` (ручная миграция, потому что BC нет). Обновлённый README как landing-page со ссылками.

### Фаза 3 — v2.1+ (расширения после стабилизации ядра)

- [ ] **События жизненного цикла модуля** — `ModuleInstalled`, `ModuleUpdated`, `ModuleRemoved`, `ModuleEnabled`, `ModuleDisabled`, `FeatureValueChanged`; хост-приложение подписывается через стандартный Laravel EventServiceProvider без модификации пакета.
- [ ] **Подпись манифестов** — опциональная криптографическая подпись `module.json` и архива (Ed25519 или GPG) для защиты канала поставки модулей заказчику; верификация в `modules:install` до распаковки.
- [ ] **Marketplace и удалённая поставка** — формат пакета (zip + manifest + signature), CLI `modules:fetch <url>` и `modules:upgrade-all` с проверкой совместимости версий, опциональный приватный реестр через HTTP/S3.
- [ ] **i18n для `settings.schema`** — `label`/`description`/`group` через ключи переводов из `Lang/{locale}/`, MoonShine-форма локализует UI по текущей локали приложения.
- [ ] **Расширенные типы фичетоглов** — `multiselect`, `json`, `file`, `relation` (с авто-резолвом через DI), валидация через Laravel-валидатор поверх JSON Schema.
- [ ] **CLI для разработчика модулей** — `modules:doctor` (диагностика манифеста, зависимостей, структуры папок), `modules:diff <name> <version>` (что изменится при update), `modules:pack <name>` (zip-сборка модуля с опциональной подписью).
- [ ] **Composer-зависимости zip-модулей** — разрешить модулю из отдельного repo/zip поставлять собственный `composer.json` как декларацию runtime-зависимостей, но устанавливать их только через root Composer host-приложения в общий `vendor`. `modules:install` должен читать поддерживаемый subset (`require` на первом этапе), валидировать conflicts до включения модуля, показывать или выполнять `composer require ...`, не применять автоматически `require-dev`, `autoload`, `scripts`, `plugins` и `repositories`, а также фиксировать связь `module → packages` для будущих `update/remove`.
