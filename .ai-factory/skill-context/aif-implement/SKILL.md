### Filesystem Atomicity and Cleanup
**Source**: 2026-05-25-13.45, 2026-05-25-14.56, 2026-05-25-17.30, 2026-05-25-22.45, 2026-05-26-22.15, 2026-05-27-03.09
**Rule**: Always check return values of filesystem operations (write byte count, delete result, rename result). Use `AtomicFileWriter` from `src/Support/` for atomic writes — check `src/Support/` for existing utilities before implementing new file I/O patterns. Production PHP cache must go through temp file + `fflush()` + atomic `rename()` under lock; wrap failed `require` in a typed exception with file path context. Clean up temp and lock files in the same `finally` block that creates them. Prefer `@unlink()` followed by `is_file()` re-check over check-then-unlink to avoid TOCTOU races. When `isFile()`/`isDirectory()` check is followed by a read operation (`get()`, `require`), wrap the read in try/catch — the file can disappear or become unreadable between the check and read.

### Lifecycle UseCase Safety Patterns
**Source**: 2026-05-26-02.15, 2026-05-26-22.15, 2026-05-27-01.30, 2026-05-27-03.09
**Rule**: When implementing lifecycle use cases (install, update, remove, scaffold), follow these safety patterns:
- Wrap post-filesystem persistence operations (manifest/state writes) in try/catch after copy/replace succeeds — on failure, trigger compensation (cleanup target directory, restore from backup, rewrite original state document) and re-throw as typed exception with original as `previous`.
- In catch blocks that perform cleanup, guard against double-fault: wrap cleanup itself in nested try/catch to preserve the original exception instead of losing it to a cleanup failure. Explicitly verify which exception variable is passed as `previous` — the original `$e` must be preserved, not the cleanup/restore error variable (common copy-paste mistake).
- Deletion order: remove recoverable state first (`state.json`), then irreversible data (module directory).
- Use `registry->has()` over try/find/catch for module existence checks — avoid flow-by-exception.
- No direct PHP filesystem calls (`file_get_contents`, `file_put_contents`, `unlink`, `rmdir`, `is_dir`) in UseCase classes — delegate to infrastructure services that use injected `Illuminate\Filesystem\Filesystem`.
- When catching exceptions in commands, use the same marker interface (`ModuleExceptionInterface`) as sibling commands in the same family — do not narrow to a specific exception subclass when other commands in the group use the broader type.

### Value Object and Type Conventions
**Source**: 2026-05-26-02.15, 2026-05-26-18.30, 2026-05-26-21.30, 2026-05-27-02.45
**Rule**: When implementing VOs and typed boundaries:
- Use typed enums (backed or unit enum) instead of string literals for classification values (source types, status codes). Pair with diagnostic DTOs that capture failure context (`key` + `reason`), not bare `list<string>`.
- Replace boolean behavior flags in public method signatures with typed enums — a bool that switches method behavior (e.g., `$deletePermanently`) violates code-quality §18 and hides intent.
- When a VO requires consistent timestamps (e.g., `installed_at` + `updated_at`), provide named constructors (`initialState()`, `updatedFrom()`) that guarantee timestamp coherence — do not expose a public constructor that accepts independent timestamp arguments.
- Collection methods (`all()`, `loadOrder()`) on snapshot/registry VOs must return deterministic order — never expose raw map iteration that depends on insertion order.
- Avoid negative boolean parameters and properties (`$disabled`, `$noBackup`) — use positive forms (`$enabled`, `$deletePermanently`).
- Keep DTOs and VOs free of service dependencies and behavior methods. If a data-carrier needs cleanup or side effects (e.g., `PreparedSource::cleanup()`), extract that responsibility to the service that created the VO.
