### Test Stub Planning for New Classes
**Source**: 2026-05-24-12.30, 2026-05-25-01.25, 2026-05-27-02.45
**Rule**: When planning new VO or service classes, always include a companion test task covering: construction from valid data, round-trip serialization (`toArray()`/`fromArray()`), and at least one error path. When planning new structural conventions (e.g., new namespace boundary, new class suffix pattern), include an architecture test task to enforce the convention via Pest `arch()` or reflection-based assertions. When planning new Artisan commands, include architecture test tasks for command dependency boundaries (commands → UseCases + Contracts only).

### Architecture Splitting Guidance
**Source**: 2026-05-24-21.30, 2026-05-25-13.45, 2026-05-25-17.30, 2026-05-26-21.30, 2026-05-26-22.15, 2026-05-27-02.45
**Rule**: When a planned class would handle 3+ distinct responsibilities, split it into role-based classes upfront during planning — do not defer decomposition to a future refactoring task. In a pre-1.0 or breaking version, plan to remove legacy sugar and dual code paths rather than maintaining backwards compatibility. When adding validation for structurally independent subtrees (e.g., top-level manifest vs settings), plan separate validator classes. `src/Support/` is the canonical location for cross-cutting infrastructure utilities — plan new shared utilities there, not alongside their first consumer. When planning to extract infrastructure from a service class, include a task for extracting the orchestration (UseCase) as well — commands should not call infrastructure directly. When planning Artisan commands, verify chosen option names against Laravel reserved names (`--force` is used by `ConfirmableTrait`). Plan `ConfirmableTrait` for all destructive commands (install, update, remove) at design time. Plan boolean behavior flags as enums from the start — a bool that switches method behavior (e.g., `$deletePermanently`) should be a typed enum. Plan DTOs/VOs as pure data carriers — if a VO needs cleanup or side effects, plan a companion service method.

### Contract-First Planning
**Source**: 2026-05-26-18.30
**Rule**: When planning features that introduce new service boundaries, include a "define interfaces/contracts" task before the implementation tasks. Contracts should be reviewed and finalized before writing concrete implementations. This prevents downstream rework when the API surface turns out to be wrong — a 30-step refactoring of ~40 files was caused by skipping contract design review.

### Interface and API Design Planning
**Source**: 2026-05-27-02.30
**Rule**: When planning interfaces, check for semantic-only method distinctions — if two methods will always return identical data by construction invariant, plan one method. Prefer names describing _what_ is returned (`all()`) over _how_ it's computed or ordered (`loadOrder()`), when the ordering is guaranteed by the data structure's construction.

### Provenance and JSON Contract Planning
**Source**: 2026-05-29-17.20, 2026-05-29-18.30, 2026-06-01-14.39
**Rule**: When planning source-origin, state, or manifest parsing work, include tasks for: whole-object provenance rewrite when installed bytes change; validated value objects for primitive domain values such as checksums; constructor and `fromArray` enforcement of cross-field invariants; closed-key validation for external JSON; explicit `null` vs missing-key tests using `array_key_exists()` semantics; and docs/AI-context sweeps for runtime narrowing or persisted format changes.
