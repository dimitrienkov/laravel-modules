### PHP Validation Gotchas
**Source**: 2026-05-24-22.00, 2026-05-25-00.30
**Rule**: When fixing validation bugs involving arrays, check for the `array_is_list([])` edge case — it returns `true` in PHP, so any "must be associative array" validation needs explicit empty-array handling. When fixing string length validation, verify whether `strlen()` should be `mb_strlen($v, 'UTF-8')` (bytes vs characters). When fixing normalization bugs, prefer explicit validation errors over silent coercion (`array_unique()`, type casting) — silent normalization hides manifest errors from users.

### Lifecycle Error Handling Patterns
**Source**: 2026-05-26-22.15, 2026-05-27-03.09
**Rule**: When fixing bugs in lifecycle use case code:
- Check catch blocks for double-fault potential — if cleanup operations can also throw, wrap them in nested try/catch to preserve the original exception.
- Replace flow-by-exception patterns (`try { find() } catch { ... }`) with explicit existence checks (`has()`) when checking module existence.
- In cleanup/error paths doing filesystem operations, prefer try-then-verify (`delete() || !isFile()`) or try/catch over check-then-act (`isFile()` then `delete()`) to avoid TOCTOU races.
- When `isFile()`/`isDirectory()` check is followed by a read operation (`get()`, `require`), wrap the read in try/catch — the file can disappear between the check and the read.
