## Rules

### Multi-Module Test Scenarios for Pipeline
**Source**: 2026-05-24-22.00
**Rule**: When creating test plans for loader pipeline changes, always include scenarios with 2+ enabled modules plus at least 1 disabled module. Single-module tests cannot detect loop ordering bugs (loaders-first vs modules-first iteration) or enabled-state filtering bugs. Verify that expected loader execution order is asserted per-module, not just per-loader.

### Lifecycle Test Setup Convention
**Source**: 2026-05-26-23.30
**Rule**: When creating QA test plans for lifecycle features, reference the `CreatesLifecycleEnvironment` trait from `tests/Support/` as the standard setup mechanism. Test scenarios should verify that new UseCase dependencies are represented in the trait's factory methods. Flag any test file that inlines lifecycle infrastructure construction instead of using the trait.

### Test Failure Injection Convention
**Source**: 2026-05-27-01.30, 2026-05-27-03.09
**Rule**: When designing test scenarios for failure paths, specify mock-based failure injection — never physical side effects (directory deletion, file corruption) that trigger real PHP warnings. For `final` infrastructure classes (e.g., `LocalFilesystem`), specify mocking the underlying `Illuminate\Filesystem\Filesystem` (non-final) and composing via the `final` wrapper to test failure paths without bypassing finality.

### State Parsing and Determinism Scenarios
**Source**: 2026-05-29-18.30, 2026-06-01-14.39
**Rule**: When creating QA test plans for state/provenance parsing, include scenarios for unknown source keys, explicit `null` checksum, scalar/list `source`, empty or whitespace `installed_version`, empty `kind`, zip-without-checksum, and local-with-checksum. Do not assert cleanup behavior with wall-clock checks like `filemtime() > time() - 5`; compare before/after artifact sets instead.

### Module-Aware Generator Regression Matrix
**Source**: 2026-06-02-20.50, 2026-06-02-22.33
**Rule**: QA plans for module-aware generators must include both sides of each text substitution: assert the module-scoped reference is present and the host reference is absent. Cover view namespace repointing, listener event repointing, form request namespace repointing, model/factory references, explicit-FQCN inputs that should remain untouched, `--module` plus `--path` conflict rejection, and the full command rebind matrix derived from the provider source list.

### Diagnostic Logging Terminal Scenarios
**Source**: 2026-06-02-12.21, 2026-06-02-13.30
**Rule**: QA plans for lifecycle diagnostics must include happy-path spy assertions for exactly one `started`, exactly one `succeeded`, and zero `failed` events, plus failure-path assertions for exactly one terminal `failed` event that carries the `Throwable`. Include a gating-matrix scenario where ERROR-level events still log at the highest threshold with the category disabled.

### Architecture Selector Scenarios
**Source**: 2026-06-02-13.30
**Rule**: When new enums, VOs, or helper types are added under a namespace covered by an architecture `toImplement()` or suffix invariant, include a test-plan item to verify selectors ignore or include the new type intentionally. Do not relocate a type away from its natural namespace just to satisfy an over-broad arch selector.
